# README.md 更新规范

## 文档定位

README.md 是**面向最终用户的产品介绍文档**，目的是让用户快速了解产品功能和使用方法。

## 核心原则

1. **用户视角**：从用户角度描述功能，不涉及技术实现
2. **功能导向**：说明产品能做什么，如何使用
3. **简洁明了**：每个功能用 2-3 句话说明
4. **保持更新**：新功能开发完成后及时添加到对应章节

## 内容范围

### ✅ 应该包含的内容

1. **项目简介**
   - 产品是什么
   - 核心亮点
   - 目标用户

2. **功能特性**
   - 按功能模块分类
   - 每个功能的简要说明
   - 使用场景示例

3. **使用指南**
   - 安装方法
   - 快速开始
   - 常见问题

4. **其他信息**
   - 开源协议
   - 贡献方式
   - 联系方式

### ❌ 不应该包含的内容

1. **版本信息**
   - ❌ 更新日志
   - ❌ 版本历史
   - ❌ 修复记录
   - ❌ 发布日期

2. **技术细节**
   - ❌ 技术架构
   - ❌ API 文档
   - ❌ 代码结构
   - ❌ 开发指南

3. **开发信息**
   - ❌ 构建脚本
   - ❌ 测试方法
   - ❌ 依赖管理
   - ❌ 开发环境配置

## 更新流程

### 何时更新

当满足以下条件时，需要更新 README.md：

1. **新增功能模块**（如：极客模式、进程冲突检测）
2. **新增子功能**（如：预设编辑、参数验证）
3. **新增用户可见特性**（如：快捷键支持、拖拽排序）
4. **功能状态变更**（如：开发中 → 已完成）

### 不需要更新的情况

- 问题修复（bug fix）
- 界面优化（UI 改进）
- 性能提升（性能优化）
- 技术改进（代码重构）
- 开发工具更新

### 更新步骤

1. **定位章节**
   - 找到功能所属的模块章节
   - 如果是新模块，在合适位置添加新章节

2. **编写内容**
   - 使用用户能理解的语言
   - 避免技术术语
   - 每个功能 2-3 句话
   - 说明功能是什么、有什么用

3. **保持格式**
   - 遵循现有的格式和风格
   - 使用统一的标题层级
   - 保持列表格式一致

4. **检查质量**
   - 确认无技术术语
   - 确认无版本信息
   - 确认描述清晰易懂
   - 确认格式统一

## 功能描述规范

### 标题格式

```markdown
### [序号]. [图标] [功能名称]

简要介绍（1-2 句话）

**核心功能**
- 功能点 1
- 功能点 2
- 功能点 3

**使用场景**
- 场景描述
```

### 描述要点

1. **功能名称**：简洁明了，用户能理解
2. **简要介绍**：1-2 句话说明功能用途
3. **核心功能**：列出主要功能点（3-5 个）
4. **使用场景**：说明什么时候用、解决什么问题

### 语言风格

**推荐用语**：
- ✅ "一键启动"
- ✅ "自动检测"
- ✅ "快速切换"
- ✅ "实时显示"
- ✅ "智能识别"

**避免用语**：
- ❌ "通过 API 调用"
- ❌ "使用 React 组件"
- ❌ "基于 Python 实现"
- ❌ "调用后端接口"
- ❌ "重构了代码"

## 检查清单

更新 README.md 后，确认：

- [ ] 新功能已添加到对应章节
- [ ] 使用用户语言，无技术术语
- [ ] 格式与现有内容保持一致
- [ ] 没有添加版本号、日期等更新信息
- [ ] 没有添加技术实现细节
- [ ] 每个功能描述清晰易懂
- [ ] 列表格式统一
- [ ] 标题层级正确

## 维护建议

1. **定期审查**
   - 每次发布新版本后检查 README
   - 确保功能描述与实际一致
   - 删除已废弃的功能

2. **保持简洁**
   - 避免过度详细的描述
   - 重点突出核心功能
   - 复杂功能可链接到详细文档

3. **用户反馈**
   - 根据用户反馈调整描述
   - 补充常见使用场景
   - 优化不清晰的表述

## 与更新日志的关系

```
更新日志（更新日志/更新日志_[版本号].md）
    ↓
  筛选新功能
    ↓
同步到 README.md
    ↓
  只保留功能描述
  删除版本、日期、技术信息
```

**关键区别**：

| 维度 | 更新日志 | README.md |
|------|---------|-----------|
| 目的 | 告知用户本版本的变化 | 介绍产品的所有功能 |
| 内容 | 新增、修复、优化 | 功能列表和使用说明 |
| 时效 | 针对特定版本 | 持续更新，反映当前状态 |
| 详细度 | 列出所有变更 | 只保留核心功能 |
| 版本信息 | 包含版本号和日期 | 不包含版本信息 |

## 常见问题

### Q: 修复了一个 bug，需要更新 README 吗？
A: 不需要。bug 修复只记录在更新日志中，README 只记录功能。

### Q: 优化了界面，需要更新 README 吗？
A: 不需要。界面优化只记录在更新日志中，除非是新增了用户可见的功能。

### Q: 新增了一个开发工具，需要更新 README 吗？
A: 不需要。开发工具不是用户功能，不记录在 README 中。

### Q: 功能从"开发中"变为"已完成"，如何更新？
A: 删除"（开发中）"标记，补充完整的功能描述。

### Q: 如何判断是否需要更新 README？
A: 问自己：这个变更是否让用户能做之前做不到的事？如果是，就需要更新。
